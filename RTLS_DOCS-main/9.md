## 1. Project Overview

This project is a reproduction and evaluation of the RTLS paper, which presents a novel approach to path planning for robotic manipulators. The core idea is:

1. **Learn a latent representation** of the robot's state space using a Variational Autoencoder (VAE) trained on randomly sampled feasible robot poses.
2. **Learn a collision classifier** that operates on the latent space, predicting whether a given latent configuration collides with a given cylindrical obstacle.
3. **Plan paths** by performing gradient-based optimisation (Activation Maximisation) in the latent space, combining three loss terms:
   - **L_goal**: Distance between the decoded end-effector position and the target position
   - **L_prior**: Encourages the latent code to stay near the training distribution (standard normal prior)
   - **L_collision**: Pushes the latent code away from collision states using the learned classifier

The approach is evaluated on a simulated **Franka Emika Panda** 7-DoF robot arm with cylindrical obstacles.

### What This Codebase Implements

This codebase follows the author's implementation as described in the paper. It includes:
- VAE training on free-space robot configurations
- Collision classifier training on labeled collision data
- Path planning evaluation using gradient-based latent-space optimisation
- MoveIt/ROS simulation for ground-truth collision checking
- Geometric capsule-based collision checking (from the author's `/sim` scripts)

### Key Modification from Author's Suggestion

After corresponding with the paper's author, a critical modification was made:
- **Before**: The trained collision classifier was used for both planning AND validation of planned paths.
- **After (current)**: The trained collision classifier is used only during **planning** (Stage 1) to provide differentiable gradients. For **validation** (Stage 2), the ground-truth geometric collision checker (`Robo3D.check_for_collision` from the `/sim` folder, cloned from the author's codebase) is used.

This two-stage approach is the author-recommended evaluation methodology.

## 3. Architecture Details

### VAE Architecture (Paper Table I vs Code)

| Component | Paper Spec | Code Implementation | Match? |
|-----------|-----------|---------------------|--------|
| **Encoder input dim** | 10 (7 joints + 3 EE) | 10 (`input_dim` in config) | YES |
| **Encoder output dim** | 7x2 (mu + logVar) | 7x2 (mu via `fc21`, logVar via `fc22`) | YES |
| **Decoder input dim** | 7 (latent) | 7 (`latent_dim` in config) | YES |
| **Decoder output dim** | 10 | 10 (`input_dim` in config) | YES |
| **Classifier input dim** | 11 (7 latent + 4 obs) | 11 (`latent_dim + obs_dim`) | YES |
| **Classifier output dim** | 1 (logit) | 1 (via `fc42`) | YES |
| **Hidden layers** | 4 | 4 (`num_hidden_layers` in config) | YES |
| **Units per layer** | 2048 | 2048 (`units_per_layer` in config) | YES |
| **Hidden activation** | ELU | ELU (throughout) | YES |

### Encoder Detail

```
Input x (10D) -> fc1(10->2048) -> ELU -> fc_encoder[0](2048->2048) -> ELU ->
fc_encoder[1](2048->2048) -> ELU -> fc_encoder[2](2048->2048) -> ELU ->
                                     fc21(2048->7) [mu, no activation]
                                     fc22(2048->7) [softplus+1e-5 -> var -> sqrt -> std]
z = mu + eps * std  (reparameterization trick)
logVar = log(std^2)  (computed for KL divergence)
```

**Variance parameterization note**: Uses `softplus` to ensure positive variance (not the more common direct log-variance parameterization). This is a numerically stable alternative.

### Decoder Detail

```
Input z (7D) -> fc31(7->2048) -> ELU -> fc_decoder[0](2048->2048) -> ELU ->
fc_decoder[1](2048->2048) -> ELU -> fc_decoder[2](2048->2048) -> ELU ->
fc41(2048->10) [no activation, linear output]
```

### Collision Classifier Detail

```
Input [z(7), obs(4)] = 11D -> fc32(11->2048) -> [ELU -> fc_obs[0](2048->2048)] ->
[ELU -> fc_obs[1](2048->2048)] -> [ELU -> fc_obs[2](2048->2048)] ->
ELU -> fc42(2048->1) [logit output]
```

**Activation ordering note**: The classifier uses a "pre-activation" pattern (`fc(ELU(h))`) in the hidden loop, whereas the encoder/decoder use a "post-activation" pattern (`ELU(fc(h))`). Despite the different coding style, the mathematical function is equivalent — there is an ELU activation between every pair of adjacent linear layers.

### Panda Robot DH Parameters

The code uses **Modified Denavit-Hartenberg** parameters matching the official Franka Emika specification:

| Joint | a (r) | d | alpha (deg) | min_theta (deg) | max_theta (deg) |
|-------|-------|------|-------------|-----------------|-----------------|
| 1 | 0 | 0.333 | 0 | -166 | 166 |
| 2 | 0 | 0 | -90 | -101 | 101 |
| 3 | 0 | 0.316 | 90 | -166 | 166 |
| 4 | 0.0825 | 0 | 90 | -176 | -4 |
| 5 | -0.0825 | 0.384 | -90 | -166 | 166 |
| 6 | 0 | 0 | 90 | -1 | 215 |
| 7 | 0.088 | 0 | 90 | -166 | 166 |

**End-effector offset**: `[0, 0, 0.107 + 0.0584]` = `[0, 0, 0.1654]` (0.107m flange + 0.0584m fingertip).

All DH parameters and joint limits match the official Franka documentation exactly.

### Robot Capsule Colliders

The robot is approximated by **11 capsules** (1 base + 10 across 7 joints):

| Location | Count | Radius | Description |
|----------|-------|--------|-------------|
| Base | 1 | 0.06m | Small capsule near robot base |
| Joint 1 | 1 | 0.06m | Vertical capsule from base to joint 1 |
| Joint 2 | 1 | 0.06m | Short capsule around joint 2 |
| Joint 3 | 1 | 0.06m | Upper arm link |
| Joint 4 | 1 | 0.06m | Elbow capsule |
| Joint 5 | 2 | 0.06m, 0.025m | Forearm link + side housing |
| Joint 6 | 1 | 0.05m | Wrist capsule |
| Joint 7 | 2 | 0.04m, 0.03m | Flange + fingertip |

### Obstacle Modeling

Obstacles are modeled as **vertical capsules** (cylindrical pillars with hemispherical caps):
- Parameters: `[x, y, h, r]` — position (x, y), height (h), radius (r)
- Capsule extends from `[x, y, 0]` (floor) to `[x, y, h]` (top)
- Always vertical, rooted at ground plane (z=0)

#### VAE Loss Function

```python
MSE = F.mse_loss(xPrime, x, reduction='sum') / batch_size   # per-sample average sum-of-squares
KL  = -0.5 * mean(sum(1 + logVar - mu^2 - exp(logVar)))     # Gaussian KL divergence
Loss = GECO.loss(MSE, KL)  # = KL + lambda * MSE  (lambda adapted by GECO)
```

#### VAE Evaluation Metrics (during training)

- **Posterior sample consistency**: Encode validation data -> decode -> FK from decoded joints -> compare FK EE position to decoded EE position
- **Prior sample consistency**: Sample from N(0,I) -> decode -> same FK consistency check
- **AM AUC**: Gradient descent in latent space toward target EE positions, success rate vs. distance threshold, AUC up to 0.1m

## 5. Evaluation Pipeline

### Overview

The evaluation pipeline (`evaluate_planning.py`) implements a **two-stage** approach:

```
STAGE 1 (Planning):  Gradient-based optimisation in latent space
                     Uses LEARNED collision classifier (differentiable)

STAGE 2 (Validation): Ground-truth collision checking on every waypoint
                      Uses GEOMETRIC capsule-based Robo3D.check_for_collision
```

### Planning Algorithm (Stage 1) — Step by Step

**Initialisation:**
1. Normalise start state `x_start = (q_start, e_start)` using training mean/std
2. Encode start state: `z_init, mu, _ = model.encoder(x_start_norm)` — uses sampled `z`, not `mu`
3. Create optimisable latent variable: `z = z_init.clone().detach().requires_grad_(True)`
4. Create Adam optimiser: `optimizer = Adam([z], lr=planning_lr)`
5. Normalise obstacles using collision dataset mean/std
6. Set initial lambdas from args

**Per-step optimisation loop (up to max_steps=300):**

```
Step A: optimizer.zero_grad()

Step B: Decode latent to configuration
        x_decoded_norm = model.decoder(z)
        x_decoded = denormalise(x_decoded_norm)
        q_decoded = x_decoded[:, :7]    (joint angles)
        e_decoded = x_decoded[:, 7:10]  (end-effector xyz)

Step C: L_goal = ||e_decoded - e_target||_2   (Euclidean distance, un-normalised)

Step D: L_prior = 0.5 * sum(z^2)   (=-log p(z) for standard normal, up to constant)

Step E: L_collision = 0.0
        For each obstacle:
            logit = model_obs.obstacle_collision_classifier(z, obs_tensor)
            p_collision = sigmoid(logit / temperature)
            L_collision += -log(1 - p_collision + 1e-8)
        L_collision /= num_obstacles   (AVERAGED, not summed)

Step F: GECO lambda update (if enabled):
        C_prior = L_prior - tau_prior_goal
        C_collision = L_collision - tau_obs_goal
        C_prior_ma = alpha_ma_prior * C_prior_ma + (1 - alpha_ma_prior) * C_prior
        C_collision_ma = alpha_ma_obs * C_collision_ma + (1 - alpha_ma_obs) * C_collision
        lambda_prior *= exp(alpha_geco * C_prior_ma)   (clipped to [1e-6, 1000])
        lambda_collision *= exp(alpha_geco * C_collision_ma)

Step G: L_total = L_goal + lambda_prior * L_prior + lambda_collision * L_collision

Step H: Record waypoint {q_decoded, e_decoded, step} in decoded_path

Step I: Early termination if dist_to_goal < success_threshold

Step J: L_total.backward(); optimizer.step()
```

### Path Validation (Stage 2) — Step by Step

```python
def validate_path_with_geometric_checker(decoded_path, obstacles_raw, robo3d):
    for each waypoint in decoded_path:
        q_rad = waypoint['q']           # joint angles in radians
        q_deg = np.degrees(q_rad)       # convert to degrees for Robo3D
        collision = robo3d.check_for_collision(q_deg, obstacles_xyhr)
        if collision:
            record collision at this waypoint
    return (is_collision_free, num_collisions, collision_waypoints)
```

**Important**: Every single decoded waypoint (one per optimisation step) is checked — there is **no interpolation or subsampling** between waypoints.

### Success Criteria

A scenario is considered successful if and only if:
```
success = goal_reached AND is_collision_free
```
Where:
- `goal_reached`: EE distance to target dropped below 0.01m (1cm) during planning
- `is_collision_free`: Zero waypoints collide per the geometric checker

## 7. Geometric Collision Checking System

### How Collision Checking Works

The collision checking pipeline in the `sim/` folder:

```
Input: joint angles (degrees) + obstacles [x, y, h, r]
  |
  v
1. Update robot state (clamp joints to limits, apply z-rotations)
  |
  v
2. get_colliders(): Transform all 11 capsule endpoints to world space
   - For each joint's capsules, local-to-world transform via parent chain
   - Uses recursive Object3D.transform_point()
  |
  v
3. Construct obstacle capsules from [x, y, h, r]
   - Each obstacle: Capsule(radius=r, point1=[x,y,0], point2=[x,y,h])
  |
  v
4. For each (robot_capsule, obstacle_capsule) pair:
   a. dist3d_segment_to_segment(s1, s2)  -- classic closest-point algorithm
   b. capsule_dist = max(0, segment_dist - radius1 - radius2)
  |
  v
5. min_dist = minimum capsule distance across all pairs
   collision = (min_dist == 0)
```

### Segment-to-Segment Distance Algorithm

Uses the well-known algorithm from `geomalgorithms.com`:
- Parameterises S1 as `s1.point1 + sc * u` and S2 as `s2.point1 + tc * v`
- Solves for `(sc, tc)` minimising distance
- Handles degenerate cases: parallel segments, parameter clipping to [0,1]

### Two FK Implementations

The codebase has **two separate FK implementations**:

1. **NumPy/SciPy FK** (via Object3D hierarchy): Used by `Robo3D` for collision checking. Recursively walks the transform hierarchy. Not differentiable.

2. **PyTorch Tensor FK** (`Panda.FK()`): Used for training evaluation and configuration generation. Batched homogeneous transform chain computed in reverse order. **Differentiable** — enables gradient flow through FK.

### Important: No Self-Collision Checking

The geometric collision system (Robo3D) only checks robot capsules against **external obstacles**. There is no robot-to-robot capsule self-collision checking. Self-collision is only checked during MoveIt-based scene generation.

## 10. Comprehensive Codebase Review — Identified Issues & Deviations

### Critical Deviations from Paper

#### Issue 1: Training Batch Size — 4096 vs Paper's 256

- **Paper** (Section IV-D): "The model is trained using a batch size of 256"
- **Code**: `batch_size = 4096` in both `config/vae_config/panda_10k.yaml` and `config/vae_obs_config/panda_10k.yaml`
- **Magnitude**: 16x larger than paper specification
- **Impact**: Larger batch sizes produce smoother gradients but can lead to sharper minima with poorer generalisation. This directly affects the quality of the learned latent space and collision classifier. The GECO reconstruction target (`g_goal = 0.0008`) may also need to be adjusted since MSE loss is normalised by batch size.
- **Location**: `config/vae_config/panda_10k.yaml` line 9, `config/vae_obs_config/panda_10k.yaml` line 4

#### Issue 2: Obstacle Min Distance from Base — Unknown Paper Value

- **Paper** (Section IV-B): "we sample a distance to origin L" — no range specified
- **Code** (`evaluate_planning.py` line 79): `MIN_DIST_FROM_BASE = 0.3m`
- **Code** (`simulate_in_moveit.py` line 1221): Hardcoded `0.25m`
- **Observed effect**: Using 0.2m gives poor results; using 0.4m gives results close to the paper
- **Impact**: This is likely the single most influential parameter on the success rate. Obstacles closer to the robot base are in the most constrained part of the workspace and are hardest to avoid.
- **Status**: Need to ask the author what value they used

#### Issue 3: Collision Loss Averaging vs Summing

- **Paper** (Eq. 6): `lambda_obs * SUM_i(-log(1 - p(z, o_i)))` — **sum** over obstacles
- **Code** (`evaluate_planning.py` line 223): `L_collision = L_collision / len(obstacles_normalized)` — **average** over obstacles
- **Impact**: With a single obstacle (our primary evaluation), this makes no difference. With multiple obstacles, the collision gradient signal is diluted by 1/N, potentially under-weighting collision avoidance.
- **Location**: `src/evaluate_planning.py` line 223

### Medium-Impact Issues

#### Issue 4: Path Length in Latent Space, Not Cartesian Space

- **Paper** (Table II): "path length is normalised by dividing the actual length of the planned path by the Euclidean distance between the initial end-effector position and the target position"
- **Code** (`evaluate_planning.py` lines 135-142): Path length = sum of L2 distances between consecutive **latent** codes
- **Impact**: Does not affect success rate, but makes path length comparisons with the paper impossible
- **Location**: `src/evaluate_planning.py` lines 135-142

#### Issue 5: No Waypoint Interpolation in Collision Validation

- Every optimisation step produces one waypoint. Collision is checked at each waypoint only.
- Between consecutive waypoints, joint angles can change enough for the robot to pass through an obstacle undetected.
- **Impact**: The geometric checker may miss collisions between decoded waypoints, potentially *inflating* the collision-free rate.
- **Mitigation**: The planning LR (0.15) with Adam produces relatively small step sizes, so this effect is likely small.

#### Issue 6: Using Sampled z vs Deterministic mu

- **Code** (`evaluate_planning.py` line 174-175): `z_init, mu, _ = model.encoder(x_start_norm)` — uses sampled `z` (stochastic via reparameterization)
- **Paper**: "z_0 ~ q_phi(z | x = x_0)" — sampling from posterior, so technically consistent
- **Impact**: Adds stochasticity that may cause some plans to fail due to a bad initial sample. Using `mu` instead would give a deterministic, more stable starting point.
- **Trade-off**: Using `mu` is more stable but deviates from the paper's formulation.

### Minor / Code Quality Issues

#### Issue 7: FK In-Place Tensor Modification

- **Location**: `src/sim/panda.py` line 66
- `joint_angles_tensor *= 180.0 / TORCH_PI` modifies input tensor in-place when `rad=True`
- **Mitigation**: Callers already use `.clone()` to prevent data corruption. Not a live bug.

#### Issue 8: Stale Echo Documentation in `run_evaluation.sh`

- The echo block at the top of `run_evaluation.sh` displays different parameter values than the actual command-line arguments
- Echoed: `lr=0.1, lambda_prior=0.07, lambda_collision=0.11, temperature=1.0, tau_prior=12.0, tau_obs=7.0`
- Actual: `lr=0.15, lambda_prior=0.7, lambda_collision=0.5, temperature=3.0, tau_prior=6.0, tau_obs=2.0`
- **Impact**: Confusing for code review, no functional impact

#### Issue 9: Debug Print in Jacobian

- **Location**: `src/sim/panda.py` line 84
- `print(j)` left in the `jacob()` method
- **Impact**: Prints debug output if Jacobian is ever called. Not used in evaluation.

#### Issue 10: Typo in Method Name

- **Location**: `src/sim/object3d.py` line 45
- `inverse_transfrom_point` should be `inverse_transform_point`
- **Impact**: Only affects code that calls this method, which appears to be unused in the current pipeline.

#### Issue 11: z_rotation_matrix_tensor_batch Device Mismatch

- **Location**: `src/sim/panda.py` line 75
- `z_rotation_matrix_tensor_batch` creates tensors on CPU, then `.to(device)` is called inside the FK loop
- **Impact**: CPU-to-GPU transfers happen inside the FK loop for every joint. Minor performance overhead.

## 12. Root Cause Analysis of Performance Gap

### Gap: 80.3% (ours) vs 85.8% (paper) = 5.5 percentage points

### Eliminated Causes

- **Scene validity** — Our scenes ARE MoveIt-validated (`test_scenes.json` confirms `"moveit_validated": true`). Start/goal configs are checked for self-collision, table collision, obstacle collision, and EE height. This was initially suspected but is NOT a cause.
- **Architecture mismatch** — All model dimensions, layer counts, activations, and DH parameters match the paper exactly.
- **Evaluation metric definition** — Success = goal reached AND collision-free with 1cm threshold. Matches paper: "within a distance threshold of 1cm and without colliding with obstacles."
- **Number of scenarios** — We use 1,000 scenarios, matching the paper.

### Most Likely Causes (ranked by probability)

**1. Obstacle minimum distance from robot base (HIGH confidence)**
- Our observation that 0.2m gives ~60% and 0.4m gives results close to 85% strongly suggests the author used a value >= 0.4m during evaluation.
- The paper does not specify this parameter.
- Currently using 0.3m (evaluate_planning.py) with scenes generated using 0.25m (simulate_in_moveit.py).

**2. Training batch size 4096 vs 256 (MEDIUM confidence)**
- 16x deviation from the paper.
- Larger batch sizes can produce models that generalise differently.
- Retraining with batch_size=256 would be the definitive test.

**3. Unknown planning hyperparameters (MEDIUM confidence)**
- The paper only mentions "grid search" for GECO planning parameters without reporting the found values.
- Our values (lr=0.15, tau_prior=6.0, tau_obs=2.0, temperature=3.0) were determined by our own grid search and may differ from the author's.

**4. Alternative explanation: Author may have reported optimistic results (LOW confidence)**
- Without access to the author's evaluation script or exact configuration, we cannot rule this out.
- The author's inability to provide the evaluation script makes verification impossible.
