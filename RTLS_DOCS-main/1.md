# How Planning Scenarios Are Generated

## The Short Answer

**We don't guarantee a feasible path exists.** The evaluation measures how often the planner can find a solution when one might or might not exist.

But we do ensure:
1. `q_start` is a valid, collision-free configuration
2. `e_target` comes from a valid, collision-free configuration
3. Both start and target are not inside obstacles

---

## Step-by-Step Scenario Generation

From evaluate_planning_proper.py and new_simulate_in_moveit.py:

### Step 1: Sample q_start (Random Joint Angles)

```python
# Get joint limits (radians)
q_min_rad = robot.joint_min_limits_tensor * (π / 180)
q_max_rad = robot.joint_max_limits_tensor * (π / 180)

# Sample uniformly within joint limits
q_start = torch.rand(1, 7) * (q_max_rad - q_min_rad) + q_min_rad

# Compute end-effector position via FK
e_start = robot.FK(q_start)
```

**This guarantees**: `q_start` is within joint limits, and `e_start` is the true EE position.

---

### Step 2: Sample q_target → Get e_target

```python
# Sample another valid joint configuration
q_target = torch.rand(1, 7) * (q_max_rad - q_min_rad) + q_min_rad

# Target EE position from FK
e_target = robot.FK(q_target)
```

**This guarantees**: `e_target` is a reachable position (because we sampled valid joints).

---

### Step 3: Validate Configurations (in MoveIt version)

```python
# Check self-collision (robot hitting itself)
start_valid = collision_oracle.check_collision(q_start)
if not start_valid:
    continue  # Resample

target_valid = collision_oracle.check_collision(q_target)
if not target_valid:
    continue  # Resample

# Check EE height (not below table)
if e_start[2] < MIN_EE_HEIGHT:
    continue

if e_target[2] < MIN_EE_HEIGHT:
    continue
```

---

### Step 4: Generate Obstacles BETWEEN Start and Goal

```python
# First obstacle always on the line from start to goal
t = random.uniform(0.3, 0.7)  # 30-70% along the path
obs_pos = e_start + t * (e_target - e_start)

obs_x, obs_y = obs_pos[0], obs_pos[1]
obs_h = random.uniform(0.5, 1.0)   # Height
obs_r = random.uniform(0.05, 0.15) # Radius
```

**This creates challenging scenarios**: The obstacle is deliberately placed to block the straight-line path.

---

### Step 5: Validate Obstacle Placement

```python
# Obstacle not too close to robot base (unavoidable if too close)
dist_from_base = sqrt(obs_x² + obs_y²)
if dist_from_base < 0.25:
    continue  # Resample

# Start EE not inside obstacle
dist_start = sqrt((e_start[0] - obs_x)² + (e_start[1] - obs_y)²)
if dist_start < obs_r and e_start[2] < obs_h:
    continue  # Start inside obstacle, resample

# Target EE not inside obstacle
dist_target = sqrt((e_target[0] - obs_x)² + (e_target[1] - obs_y)²)
if dist_target < obs_r and e_target[2] < obs_h:
    continue  # Target inside obstacle, resample
```

---

### Step 6 (MoveIt version): Check Full Arm Collision

```python
# Add obstacles to simulation
collision_oracle.add_obstacles_from_array(obstacles)

# Check if robot ARM (not just EE) collides at start
start_valid = collision_oracle.check_collision(q_start)
if not start_valid:
    continue  # Robot arm hits obstacle at start

# Check if robot ARM collides at target
target_valid = collision_oracle.check_collision(q_target)
if not target_valid:
    continue  # Robot arm hits obstacle at target
```

## What's Guaranteed vs Not Guaranteed

| Property | Guaranteed? | How |
|----------|-------------|-----|
| `q_start` within joint limits | ✅ Yes | Sampled uniformly within limits |
| `q_start` is self-collision free | ✅ Yes (MoveIt) | Checked with collision oracle |
| `e_start` is above table | ✅ Yes | Height check |
| `q_target` is valid | ✅ Yes | Same checks as start |
| Start/target not inside obstacles | ✅ Yes | Geometric distance check |
| **A feasible path exists** | ❌ NO | Not checked |

---

## Why Don't We Guarantee Feasible Paths?

**Reason 1: Computational cost**

Verifying a feasible path exists requires running a motion planner (like RRT*), which takes seconds per scenario. For 1000 scenarios, this would be hours.

**Reason 2: It's a realistic evaluation**

In real robotics, you don't always know if a path exists before planning. The planner must handle infeasible cases gracefully.

**Reason 3: The paper's approach**

The paper evaluates:
- **Success rate**: What fraction of scenarios does the planner solve?
- **Goal-reaching rate**: How often does it get close to the target?
- **Collision-free rate**: How often does it avoid obstacles?

Some scenarios may be impossible. The success rate reflects both planner capability AND scenario difficulty.

## Loading Pre-Generated Scenarios

For reproducibility, scenarios can be saved and loaded:

```python
if loaded_scenarios:
    # Load pre-generated scenario
    scenario = loaded_scenarios['scenarios'][i]
    q_start = torch.tensor(scenario['q_start'])
    e_start = torch.tensor(scenario['e_start'])
    e_target = torch.tensor(scenario['e_target'])
    obstacles = scenario['obstacles']
```

This ensures different planners are evaluated on identical scenarios.
